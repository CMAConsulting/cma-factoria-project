\section{Resumen Ejecutivo}

\subsection{¿Qué es SYN FactorIA?}

SYN FactorIA es una plataforma de desarrollo de software que integra agentes de inteligencia artificial especializados en cada fase del ciclo de vida del software. Utilizando los más recientes avances en modelos de lenguaje grandes (LLMs) y arquitecturas de agentes autónomos, SYN FactorIA transforma la manera en que se construyen aplicaciones empresariales modernas.

\subsection{Propuesta de Valor}

\begin{itemize}
    \item \textbf{Reducción de costos del 40-60\%} en proyectos de desarrollo mediante automatización inteligente
    \item \textbf{Time-to-market acelerado} hasta un 50\% más rápido que metodologías tradicionales
    \item \textbf{Consistencia arquitectónica} garantizada mediante contratos OpenAPI y patrones predefinidos
    \item \textbf{Escalabilidad técnica} con arquitectura de microfrontends y microservicios
\end{itemize}

\subsection{Mercado Objetivo}

\begin{itemize}
    \item Empresas de consultoría de software que buscan diferenciarse
    \item Equipos de desarrollo que requieren velocidad sin sacrificar calidad
    \item Organizaciones que buscan adoptar prácticas de DevOps e IA generativa
\end{itemize}

\subsection{Arquitectura Vertical Slice}

SYN FactorIA implementa el patrón Vertical Slice Architecture para maximizar la eficiencia de los agentes de IA:

\subsubsection{¿Qué es Vertical Slice?}

Vertical Slice Architecture es un patrón arquitectónico donde cada ``slice'' o ``rebanada'' atraviesa toda la aplicación, desde la capa de presentación hasta la base de datos, encapsulando una funcionalidad completa del negocio. A diferencia de las arquitecturas tradicionales por capas (UI / Negocio / Datos), Vertical Slice agrupa el código por casos de uso o features.

\subsubsection{Beneficios para Agentes de IA}

\begin{itemize}
    \item \textbf{Contexto limitado por slice}: Cada slice contiene solo el código relevante para una funcionalidad, reduciendo el contexto necesario para que el agente de IA comprenda y modifique el código.
    \item \textbf{Reducción de tokens}: Al tener slices pequeños y autocontenidos, el consumo de tokens en prompts se reduce significativamente.
    \item \textbf{Foco en características}: Los agentes pueden trabajar en una feature sin necesidad de entender toda la aplicación.
    \item \textbf{Cambios localizados}: Modificaciones en un slice no afectan a otros, minimizando el riesgo de efectos secundarios.
\end{itemize}

\subsubsection{Estructura Vertical Slice en SYN FactorIA}

\begin{verbatim}
apps/backend/[microservicio]/
├── src/main/java/org/cma/factoria/[slice]/
│   ├── endpoint/          # REST Resource (JAX-RS)
│   │   └── *Resource.java
│   ├── service/           # Lógica de negocio
│   │   └── *Service.java
│   ├── repository/        # Acceso a datos
│   │   └── *Repository.java
│   └── model/            # Modelos específicos del slice
│       ├── *Request.java
│       └── *Response.java
└── src/main/java/org/cma/factoria/shared/
    └── model/            # Modelos compartidos (OpenAPI)
\end{verbatim}

\subsubsection{Integración con Agentes}

Los agentes de SYN FactorIA están configurados para:

\begin{enumerate}
    \item Identificar el slice relevante para la tarea solicitada
    \item Leer solo los archivos del slice seleccionado
    \item Generar código dentro del contexto del slice
    \item Validar que los cambios no afectan otros slices
\end{enumerate}

Esta estrategia permite que los agentes trabajen de manera más eficiente, reduciendo costos de API y mejorando la calidad del código generado al tener un contexto más preciso y limitado.

\subsection{Reglas de los Agentes}

Los agentes en SYN FactorIA siguen reglas estrictas para garantizar calidad y consistencia:

\begin{enumerate}
    \item \textbf{No modificar especificaciones}: Los agentes de implementación nunca alteran SPEC.md, solo lo leen
    \item \textbf{Usar scripts comunes}: Scripts/commons/ proporciona funciones reutilizables para logging, validación y esperas
    \item \textbf{Soporte de perfiles}: Todos los scripts deben aceptar -{}-profile para configuración por ambiente
    \item \textbf{Compilación antes de entrega}: Ejecutar build/typecheck antes de reportar completitud
    \item \textbf{Documentación}: Cambios relevantes documentados en docs/
    \item \textbf{Validación de arquitectura}: Seguir patrones de Vertical Slice para limitar contexto de agentes
\end{enumerate}

\section{Caso de Uso: SYN FactorIA en Acción}

\subsection{Escenario Típico de Desarrollo}

El flujo de trabajo typical sigue un patrón de orquetación multi-agente:

\begin{enumerate}
    \item \textbf{Recibir requerimiento}: Definición de nueva funcionalidad o corrección
    \item \textbf{Análisis (Optimizer)}: El agente investigdor analiza el código existente y genera SPEC.md con especificaciones técnicas detalladas
    \item \textbf{Implementación (Dev Senior/Backend Senior/Frontend Senior)}: Los agentes de implementación escriben código siguiendo las especificaciones y patrones del proyecto
    \item \textbf{Validación (QA Senior)}: El agente validador ejecuta pruebas adversariales, verifica builds y compliance con SPEC.md
    \item \textbf{Consolidación (Product Owner)}: El orquestador consolida cambios, actualiza documentación y registra en docs/history/
\end{enumerate}

Este flujo garantiza que cada entregable pase por análisis, implementación y validación antes de ser fusionado.

\subsubsection{Sistema de Perfiles con -{}-profile}

SYN FactorIA implementa un sistema de configuración por perfiles que permite inicializar diferentes ambientes:

\begin{verbatim}
# Cargar variables de entorno por perfil
source scripts/commons/get.sh
load_env_vars "dev"          # Carga dev.env
load_env_vars "staging"     # Carga staging.env
load_env_vars "prod"        # Carga prod.env

# Los scripts de backend/frontend aceptan --profile
./scripts/backend/local_start.sh --profile dev
./scripts/frontend/local_start.sh --profile staging
\end{verbatim}

\subsubsection{Beneficios del Sistema de Perfiles}

\begin{itemize}
    \item \textbf{Configuración por ambiente}: Cada perfil (dev, staging, prod) tiene sus propias variables
    \item \textbf{Secrets separados}: Credenciales y API keys aisladas por ambiente
    \item \textbf{Puerto y host configurables}: Endpoints específicos por ambiente
    \item \textbf{Integración con agentes}: Los perfiles se cargan automáticamente según el contexto
    \item \textbf{Documentación automática}: Cada .env genera documentación de variables disponibles
\end{itemize}

\subsection{Integración con Capas de Aplicación}

El sistema de perfiles proporciona una capa de configuración que se inyecta en:

\begin{itemize}
    \item \textbf{Backend (Quarkus)}: Variables de entorno para conexión a base de datos, puertos, URLs de APIs
    \item \textbf{Frontend (React)}: URLs de APIs, flags de feature toggles, configuraciones de ambiente
    \item \textbf{Docker}: Build args y environment variables para contenedores
    \item \textbf{Terraform/Infraestructura}: Credenciales AWS, regiones, nombres de recursos
    \item \textbf{Agentes IA}: Contexto de ambiente para generación de código específica
\end{itemize}

    \item Establecer métricas de éxito y KPIs
    \item Escalar gradualmente a más proyectos
\end{enumerate}
